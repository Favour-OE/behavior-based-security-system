What is BBSS?

BBSS (Behavior-Based Security System) is a Python package that implements continuous authentication through behavioral biometrics. It goes beyond traditional password-based authentication by monitoring user behavior patterns and flagging sessions that deviate from established baselines.

The system captures behavioral signals (typing speed, login times, command patterns, IP addresses) and builds statistical profiles. When a session deviates from this baseline, BBSS computes a risk score and triggers appropriate security responses: allow, challenge with PIN verification, or block.

Key Features

  • Continuous authentication: Validates sessions in real-time, not just at login
  • Behavioral profiling: Learns user patterns—typing speed, session timing, command sequences
  • Statistical anomaly detection: Uses z-score and IQR-based analysis
  • Risk-based responses: Smart escalation—allow safe sessions, challenge suspicious ones, block threats
  • Machine learning ready: Isolation Forest models for advanced anomaly detection
  • Runtime configuration: All settings configurable at runtime via code or environment
  • Production-ready: 88+ tests with full coverage, modular design

Quick Start

1. Basic Usage with BBSS Class

from bbss import BBSS # Initialize the engine engine = BBSS() # Register a new user result = engine.signup(username="john_doe", password="SecurePassword123!") print(result) # {"success": True, "user_id": 1, "error": None}

2. Authenticate with Behavioral Capture

# Authenticate a user result = engine.login( username="john_doe", password="SecurePassword123!", typing_time=2.14, # seconds taken to type password ip_address="192.168.1.10" ) print(result) # { # "success": True, # "session": , # "session_token": "a3f9e8c2...", # "risk_score": 10, # "decision": "ALLOW", # "anomaly_signals": [] # }

3. Handle Session Based on Decision

result = engine.login("john_doe", "SecurePassword123!", typing_time=5.0) if result["success"]: if result.get("session"): # Safe session - use the session object session = result["session"] engine.execute_command(session, "view_report") engine.end_session(session) elif result.get("requires_verification"): # Warning level - show PIN challenge print(f"Enter PIN: {result['pin']}") else: # Blocked - high risk detected print("Session blocked:", result.get("error"))

4. Using Module Functions (Alternative)

from bbss import signup, login, execute_command, end_session, Session # Register signup("jane", "Password123!") # Login result = login("jane", "Password123!", typing_time=1.5) session = result["session"] # Record commands execute_command(session, "view_report") execute_command(session, "export_data") # End session end_session(session)

Architecture

BBSS follows a modular architecture with clear separation of concerns:

Module Description
auth User registration, login, password hashing (Argon2), session management
behavior Behavioral signal capture, profile building, anomaly detection
security Risk scoring, risk classification, security response dispatcher
database SQLite database management, models, query helpers
ml Machine learning models (Isolation Forest), feature extraction
logs Security logging, audit trail management
config Runtime configuration management

Risk Scoring

Every session receives a risk score from 0–100 based on detected anomalies. The score determines the security response:

Score Range Classification Response
0–30 SAFE Grant full access immediately
31–60 WARNING Secondary verification (PIN challenge)
61–100 HIGH_RISK Block session immediately

Each anomaly signal contributes a weighted score. Default weights are configurable via RISK_WEIGHT_* settings. The total risk score is capped at 100 to prevent overflow.

Note: During the Bootstrapping stage (first 4 sessions), all risk scores are 0 since no baseline exists yet. During Warmup (sessions 5-9), risk scores are halved to reduce false positives.

Behavioral Signals

BBSS captures and analyzes the following behavioral signals during each session:

Signal Detection Method Default Weight Description
typing_time_deviation Z-score analysis 20 Deviation from user's typical typing speed
unusual_login_hour IQR analysis 25 Login time outside normal hours
unusual_login_day Pattern matching 10 Login on unusual day of week
unknown_ip IP whitelist check 20 Login from unrecognized IP address
command_count_anomaly Z-score analysis 15 Unusual number of commands for session
unknown_commands Command whitelist 10 Execution of uncommon commands
session_duration_anomaly Z-score analysis 10 Session length differs from baseline
ml_isolation_forest ML model prediction 30 Isolation Forest flags session as anomalous

Detection Algorithms

Z-score analysis measures how many standard deviations a value is from the mean:

z = (current_value - baseline_mean) / baseline_std If |z| > ZSCORE_THRESHOLD (default: 2.0), the signal is flagged as anomalous.

IQR analysis uses the interquartile range to identify outliers in time-based data:

typical_range = [Q1 - IQR_MULTIPLIER * IQR, Q3 + IQR_MULTIPLIER * IQR] If value falls outside this range, it is flagged. Default multiplier is 1.5.

Profile Lifecycle

BBSS handles new users gracefully with a three-stage profile maturity system. This prevents false positives during the initial learning period:

Stage Sessions Sensitivity Anomaly Detection
Bootstrapping 1–4 0% Disabled — all sessions treated as safe
Warmup 5–9 50% Reduced sensitivity — risk scores halved
Active 10+ 100% Full detection with normal thresholds

The system automatically transitions users between stages based on their session count. Configuration options allow customization:

MIN_SESSIONS_BOOTSTRAPPING = 5 # Sessions before warmup MIN_SESSIONS_WARMUP = 10 # Sessions before active ML_MIN_SESSIONS_FOR_TRAINING = 20 # Sessions before ML training
Tip: For testing, you can reduce these thresholds to speed up profile maturity: engine = BBSS(MIN_SESSIONS_BOOTSTRAPPING=1, MIN_SESSIONS_WARMUP=2)

BBSS Class

The main entry point for using BBSS. Instantiate this class to get access to all authentication and behavioral analysis features.

Constructor

BBSS(**config_overrides)

Initialize the BBSS engine. All configuration options can be passed as keyword arguments for per-instance customization.

# Use all defaults engine = BBSS() # Override specific settings for this instance engine = BBSS( RISK_THRESHOLD_WARNING=40, RISK_THRESHOLD_HIGH=80, ML_ENABLED=False )

Properties

Property Type Description
config Config Current configuration instance with all active settings

Methods

Method Description
signup(username, password, email?) Register a new user with secure password hashing
login(username, password, typing_time?, ip_address?, user_agent?) Authenticate user and return login result with session
execute_command(session, command_name) Record a command executed during active session
end_session(session) End an active session and save behavior data
get_user_report(user_id) Get security report including risk history for a user
verify_pin(session, pin) Verify a PIN for sessions requiring secondary auth

Module Functions

Alternative to using the BBSS class. These functions use a global singleton engine for simpler usage in smaller applications.

from bbss import ( signup, login, execute_command, end_session, configure, get_config, BBSS, Session )
Function Description
signup(username, password, email?) Register new user using global engine
login(username, password, typing_time?, ip_address?, user_agent?) Authenticate user using global engine
execute_command(session, command_name) Record command using global engine
end_session(session) End session using global engine
configure(**kwargs) Configure global singleton settings
get_config() Get current global config instance
Note: Module functions share state through a global singleton. For multi-tenant or isolated scenarios, use the BBSS class directly.

Runtime Configuration

BBSS supports three levels of configuration (in priority order):

  1. Programmatic overrides (highest priority)
  2. Environment variables
  3. Default values (lowest priority)

Option 1: Per-Instance Configuration

from bbss import BBSS # Override specific settings for this instance engine = BBSS( DATABASE_PATH="./production.db", RISK_THRESHOLD_HIGH=80, ML_ENABLED=True, LOG_LEVEL="DEBUG" ) # Access the config print(engine.config.RISK_THRESHOLD_HIGH) # 80

Option 2: Global Configuration

from bbss import configure, get_config # Configure globally before using module functions configure( LOG_DIR="./custom_logs", CONSOLE_LOGGING=False, MAX_FAILED_ATTEMPTS=3 ) # Use the config elsewhere config = get_config() print(config.MAX_FAILED_ATTEMPTS) # 3

Option 3: Environment Variables

# .env file DATABASE_PATH=./production.db RISK_THRESHOLD_WARNING=40 RISK_THRESHOLD_HIGH=80 ML_ENABLED=true LOG_LEVEL=DEBUG
Note: Programmatic configuration takes precedence over environment variables, which take precedence over defaults.

Troubleshooting

Common Issues

ImportError: No module named 'bbss'
Solution: Run pip install -e . in the project root directory to install the package in development mode.
Session blocked unexpectedly for new user
Solution: New users in bootstrapping stage (first 4 sessions) should never be blocked. Check that MIN_SESSIONS_BOOTSTRAPPING is set correctly and that the database is properly initialized.
ML model not training
Solution: Ensure ML_ENABLED=True and the user has at least ML_MIN_SESSIONS_FOR_TRAINING (default: 20) sessions. Check that the ML models directory is writable.
Config changes not taking effect
Solution: Configuration is read at engine initialization. Ensure configure() or constructor arguments are set before creating the BBSS engine instance.
Database locked errors
Solution: SQLite has limited concurrent write support. Use PostgreSQL for production multi-user scenarios. Ensure only one process writes to the database at a time.
Password verification failing
Solution: Argon2 hashing is intentionally slow. If testing feels slow, use :memory: database with reduced Argon2 parameters, not in production.

Debug Mode

Enable debug logging to diagnose issues:

from bbss import configure configure(LOG_LEVEL="DEBUG", CONSOLE_LOGGING=True) # Now all operations will print detailed logs
